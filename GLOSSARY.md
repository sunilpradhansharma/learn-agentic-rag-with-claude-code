# Glossary

Terms introduced during the course. Each term links to the lesson where it's first introduced.

---

### Agent
*(introduced in [Lesson 3](lessons/03-tiny-rag/README.md))*

A program that uses a language model to decide what actions to take in order to complete a goal, rather than executing a fixed sequence of steps. Agents can call tools, observe results, and choose subsequent actions dynamically based on context.

---

### Agentic RAG
*(introduced in [Lesson 3](lessons/03-tiny-rag/README.md))*

A RAG system in which retrieval, reasoning, and generation are controlled by one or more agents. Unlike a fixed RAG pipeline, an agentic RAG system can decide when to retrieve, what to retrieve, whether the retrieved content is sufficient, and whether to retry or reformulate a query.

---

### Chunk / Chunking
*(introduced in [Lesson 4](lessons/04-loading-chunking/README.md))*

A chunk is a contiguous segment of text extracted from a larger document. Chunking is the process of splitting documents into chunks of a controlled size before embedding them. Chunk size and overlap are design decisions that directly affect retrieval quality.

---

### CLAUDE.md
*(introduced in [Lesson 2](lessons/02-claude-md/README.md))*

A Markdown file placed at the root of a project (or in a subdirectory) that Claude Code reads automatically when it starts. It provides persistent project-level context — coding conventions, architecture notes, workflow rules — so you do not have to repeat this information in every prompt.

---

### Context window
*(introduced in [Lesson 1](lessons/01-first-file/README.md))*

The maximum amount of text (measured in tokens) that a language model can process in a single call, including both the input and the output. Content outside the context window is not visible to the model. Managing what goes into the context window is a central concern in RAG system design.

---

### Corrective RAG / CRAG
*(introduced in [Lesson 11](lessons/11-self-reflection/README.md))*

A retrieval strategy in which the system evaluates the quality of retrieved documents and, if they are deemed insufficient or irrelevant, takes a corrective action — such as reformulating the query, retrieving from a different source, or falling back to a web search — before generating an answer.

---

### Cosine similarity
*(introduced in [Lesson 5](lessons/05-embeddings-search/README.md))*

A measure of similarity between two vectors, computed as the cosine of the angle between them. Values range from -1 to 1; a value of 1 means the vectors point in the same direction. In semantic search, cosine similarity between an query embedding and a document embedding indicates how semantically related the two texts are.

---

### Cross-encoder
*(introduced in [Lesson 9](lessons/09-retrieval-quality/README.md))*

A model that takes a query and a document as a joint input and outputs a relevance score. Cross-encoders are more accurate than embedding-based similarity for ranking but are too slow to run over a full corpus; they are used as a reranker over a small candidate set retrieved by a faster first-stage retriever.

---

### Embedding
*(introduced in [Lesson 5](lessons/05-embeddings-search/README.md))*

A fixed-length vector of floating-point numbers that represents the semantic content of a piece of text. Texts with similar meaning have embeddings that are close together in vector space. Embeddings are produced by an embedding model and stored in a vector store for later retrieval.

---

### Faithfulness
*(introduced in [Lesson 8](lessons/08-ragas/README.md))*

A RAGAS metric that measures whether every claim in a generated answer is supported by the retrieved context. A faithful answer makes no claims that go beyond what the context contains. Faithfulness is distinct from correctness: an answer can be faithful to a wrong or incomplete context.

---

### Golden dataset
*(introduced in [Lesson 7](lessons/07-handrolled-evals/README.md))*

A curated set of question–answer pairs with known correct answers, used to evaluate a RAG system. Building a golden dataset is a deliberate act: questions are chosen to cover the range of behaviors you care about, and answers are verified against the source documents.

---

### Guardrails
*(introduced in [Lesson 17](lessons/17-guardrails/README.md))*

Validation and enforcement logic applied to the inputs or outputs of an LLM to ensure they meet defined safety, quality, or policy requirements. Guardrails AI is a Python framework that provides a library of validators and a runtime for applying them to LLM calls.

---

### Hallucination
*(introduced in [Lesson 6](lessons/06-naive-rag/README.md))*

Output generated by a language model that is factually incorrect or not grounded in the provided context. In RAG systems, hallucinations often occur when retrieved documents do not contain the information needed to answer a question and the model fills the gap with plausible-sounding but incorrect content.

---

### HyDE
*(introduced in [Lesson 10](lessons/10-query-rewriting/README.md))*

Hypothetical Document Embeddings. A query rewriting technique in which the LLM generates a hypothetical answer to the query, and that hypothetical answer — rather than the original query — is embedded and used for retrieval. The hypothesis is often more semantically similar to relevant documents than the original question.

---

### Hybrid search
*(introduced in [Lesson 9](lessons/09-retrieval-quality/README.md))*

A retrieval strategy that combines dense semantic search (embedding similarity) with sparse keyword search (such as BM25). The two result sets are merged and scored together, typically using Reciprocal Rank Fusion (RRF). Hybrid search often outperforms either method alone, particularly on queries that contain specific terms or identifiers.

---

### LangGraph
*(introduced in [Lesson 14](lessons/14-langgraph-basics/README.md))*

A Python library from LangChain for building stateful, graph-structured agent workflows. In LangGraph, a pipeline is modeled as a directed graph of nodes (functions) and edges (transitions). The graph can contain cycles, enabling loops such as retry and self-reflection.

---

### LLM-as-judge
*(introduced in [Lesson 8](lessons/08-ragas/README.md))*

An evaluation pattern in which a language model is used to score or compare the outputs of another language model (or the same one). The judge model is given an answer, optionally the question and retrieved context, and asked to rate quality along a defined dimension such as faithfulness or relevance.

---

### Multi-agent system
*(introduced in [Lesson 15](lessons/15-multi-agent-rag/README.md))*

An architecture in which multiple agents, each with a defined role and set of capabilities, collaborate to complete a task. Agents communicate by passing messages or shared state. Multi-agent systems are used when a task is too complex or broad for a single agent to handle reliably.

---

### Observability / Tracing
*(introduced in [Lesson 16](lessons/16-phoenix-observability/README.md))*

Observability is the ability to understand the internal state of a system from its external outputs. In LLM systems, this is typically implemented through tracing: recording the inputs, outputs, latency, and metadata of every LLM call and tool invocation. Arize Phoenix provides a UI for viewing and analyzing these traces.

---

### Prompt
*(introduced in [Lesson 1](lessons/01-first-file/README.md))*

The text input passed to a language model to elicit a response. A prompt may include instructions, examples, context, and a question or task. In a RAG system, the prompt typically includes the retrieved documents alongside the user's question.

---

### RAG (Retrieval-Augmented Generation)
*(introduced in [Lesson 3](lessons/03-tiny-rag/README.md))*

An architecture that augments a language model's response by first retrieving relevant documents from an external knowledge base and including them in the prompt. RAG allows a model to answer questions about content it was not trained on, and reduces hallucination by grounding generation in retrieved evidence.

---

### RAGAS
*(introduced in [Lesson 8](lessons/08-ragas/README.md))*

An open-source Python framework for evaluating RAG systems. RAGAS provides reference-free metrics — including faithfulness, answer relevancy, context precision, and context recall — that can be computed without human-labeled ground truth for every answer.

---

### Reranker
*(introduced in [Lesson 9](lessons/09-retrieval-quality/README.md))*

A model or scoring function applied after an initial retrieval step to re-order a candidate set of documents by relevance. Rerankers trade off speed for accuracy: they are slower than embedding similarity but produce more precise rankings, particularly for nuanced queries.

---

### Retriever
*(introduced in [Lesson 3](lessons/03-tiny-rag/README.md))*

The component of a RAG system responsible for finding relevant documents given a query. A retriever queries a vector store, keyword index, or other data source and returns a ranked list of documents to be included in the prompt.

---

### Self-reflection
*(introduced in [Lesson 11](lessons/11-self-reflection/README.md))*

A technique in which an agent evaluates its own output or intermediate reasoning and decides whether to revise it before proceeding. In RAG, self-reflection is often applied after generation: the model checks whether its answer is supported by the retrieved context and, if not, triggers another retrieval or generation step.

---

### Semantic search
*(introduced in [Lesson 5](lessons/05-embeddings-search/README.md))*

Search based on the meaning of text rather than exact keyword matches. A query is converted to an embedding and compared against document embeddings in a vector store using a similarity metric such as cosine similarity. Documents semantically related to the query are returned even if they share no words with it.

---

### Sub-agent
*(introduced in [Lesson 13](lessons/13-subagents/README.md))*

An agent that is invoked by a parent agent to handle a specific subtask. The parent agent delegates work to sub-agents and aggregates their results. This pattern decomposes complex tasks into smaller, more focused units and allows different sub-agents to specialize in different capabilities.

---

### Tool use
*(introduced in [Lesson 12](lessons/12-tool-use/README.md))*

The ability of a language model to call external functions or APIs during generation. The model is given a list of available tools with descriptions and schemas; it can choose to call a tool, observe the result, and incorporate it into its response. Tool use extends what a model can do beyond what it can reason about from its training data alone.

---

### Vector store
*(introduced in [Lesson 5](lessons/05-embeddings-search/README.md))*

A database optimized for storing and querying high-dimensional embedding vectors. Given a query vector, a vector store returns the most similar stored vectors using an approximate nearest-neighbor (ANN) index. Common vector stores include ChromaDB, Pinecone, Weaviate, and pgvector.
